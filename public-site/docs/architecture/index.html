Architecture

Two layers, one product path

Beagle OS has exactly two layers: the Beagle control plane on the Beagle host and the Beagle thin-client operating system on the endpoint. The product path remains fixed: Beagle Stream Server in the target VM, Beagle Stream Client on the endpoint, managed through Beagle Control Plane.

The two-layer model

Layer one is the Beagle control plane on the Beagle host. It integrates into the Beagle WebUI, adds the Create Beagle VM workflow, generates VM-specific artifacts, exposes a local REST API, and manages USB control, egress policy, endpoint identity, and fingerprint awareness.

Layer two is the Beagle thin-client operating system. It is Debian-based, boots directly into Beagle Stream Client, and avoids the general-purpose desktop model.

System flow

AdminWorks in the Beagle WebUI and control plane.
Control PlaneGenerates artifacts, policies, identity state, and API responses.
Target VMRuns Beagle Stream Server as the streaming server.
EndpointBoots Beagle OS and launches Beagle Stream Client.

VM-centric instead of user-centric

The VM is the streaming target and the provisioning anchor. Beagle generates USB installers and runtime profiles for a specific VM, not for a generic user account that later chooses a target.

Why only Beagle Stream Client and Beagle Stream Server

Beagle deliberately chooses one protocol, one client, and one server. This reduces operational variance, improves latency-focused testing, and makes endpoint behavior reproducible.
